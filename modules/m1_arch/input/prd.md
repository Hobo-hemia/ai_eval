# 客户服务中心 2.0 产品需求（PRD 摘要）

## 背景

我们当前的“在线工单”服务最初是为单一游戏业务设计的（见 `api.proto` v1），现在要升级为“客户服务中心 2.0”。
未来它将作为底层基建，支撑集团线300+个业务（涵盖游戏、电商周边、云服务平台等）的接入。
由于涉及历史包袱，原有的 v1 客户端（App 较旧版本）在未来一年内仍会继续发送旧版请求，不可强制要求立刻升级。

## 核心痛点与业务诉求

1. **入口爆炸**：各业务线诉求繁多，现在前端维护了十几套不同接口入口，新版要求在前端使用“统一表单”提交，后端必须能利用单一抽象载体消化。
2. **处理过程不可见**：用户抱怨提交工单后如“石沉大海”。业务方要求新版必须能追溯某个单子被谁接手、流转到了哪个组、状态变更的时间节点。
3. **多人多节点协作**：一个单子可能不仅是客服在看，商家、风控甚至外包合规团队都可能介入处理，系统必须能承载这种“多方协作”的信息。
4. **大促期间管理低效**：双11期间工单暴增，运营人员抱怨只能一个个点开处理，强烈要求能在后台一次性框选数千个单子“一键转交”或“一键关单”。
5. **合规与响应风险**：由于海外合规要求，如果一个单子距离SLA（承诺响应时间）仅剩2小时，前端业务流需要给出醒目预警。现有查询接口无法支撑这种实时风险暴露。

## 架构演进期望

- 请直接升级并产出完整的协议文件。
- 设计必须具备极高扩展性，能够灵活适应完全不同业务线的数据结构差异（比如游戏业务有 ServerID，电商业务有 OrderID），但又不能完全退化为毫无约束的 JSON String。
- 必须兼容 v1 客户端，或提供统一的向下兼容/过渡方案，千万不能让还在使用 v1 的老版本 App 直接宕机。
